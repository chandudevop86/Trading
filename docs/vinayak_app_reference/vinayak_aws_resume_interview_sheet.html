Project Summary

Vinayak is a Python trading platform evolving from a monolithic trading workspace into a production-ready AWS 3-tier architecture with separated web, application, and data responsibilities.

What The Platform Does

  • Runs live market analysis and strategy workflows
  • Supports reviewed trade execution paths
  • Integrates with Dhan broker APIs
  • Sends Telegram alerts and summaries
  • Generates reports and exports

Why It Fits AWS Well

  • Clear public web entry through ALB
  • Stateless app layer fits ECS Fargate
  • Relational trade state fits RDS PostgreSQL
  • Hot market data fits Redis caching
  • Artifacts naturally move to S3

Interview Explanation

I built a trading platform that originally behaved like a Python monolith, then separated it into a cleaner architecture called Vinayak. The target design uses Route 53 and ALB for the web tier, ECS Fargate for the application tier, and RDS plus Redis plus S3 for the data layer.

Strong Resume Language

Designed a trading platform migration from a monolithic Python workflow into an AWS-ready 3-tier architecture using ALB, ECS Fargate, RDS PostgreSQL, ElastiCache Redis, S3, and Secrets Manager.

Connection Flow

  1. User traffic enters through Route 53 and ALB.
  2. ALB routes requests to ECS Fargate tasks running Vinayak.
  3. Vinayak processes admin, workspace, strategy, and execution flows.
  4. Persistent business data is stored in RDS PostgreSQL.
  5. Hot cache and repeated market context are stored in Redis.
  6. Reports and exported files are stored in S3.
  7. Secrets such as DB, Dhan, and Telegram credentials are injected from Secrets Manager.
Current project note: some migration paths still reuse legacy Trading modules and some local-file fallbacks still exist, but the architecture direction now clearly matches a 3-tier AWS model.
