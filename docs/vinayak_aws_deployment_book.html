Executive Summary

Vinayak should run as a clean AWS three-tier deployment:

  1. Web tier: public-facing entry point and reverse proxy
  2. Application tier: FastAPI, trading orchestration, execution, and observability
  3. Data tier: PostgreSQL as the durable system of record

This separation improves reliability, security, and production readiness for controlled live validation.

Chapter 1: AWS 3-Tier Architecture

Web Tier

Recommended: EC2 with Nginx behind an Application Load Balancer.

Responsibilities:

  • receive public traffic
  • reverse proxy to the app tier
  • avoid storing broker or DB credentials

Application Tier

Recommended: private EC2 running vinayak.api.main:app.

Responsibilities:

  • FastAPI
  • strategy evaluation
  • broker integration
  • observability APIs
  • operator actions

Data Tier

Recommended: Amazon RDS PostgreSQL.

Responsibilities:

  • reviewed trades
  • executions
  • audit logs
  • application state

Optional Cache Tier

Recommended: Amazon ElastiCache Redis.

Chapter 2: Security Groups and Network Layout

Use:

  • 2 public subnets
  • 2 private app subnets
  • 2 private data subnets

Security groups:

  • sg-alb: inbound 80/443 from internet
  • sg-web: inbound only from sg-alb, SSH only from your IP
  • sg-app: inbound 8000 only from sg-web or sg-alb
  • sg-db: inbound 5432 only from sg-app
  • sg-redis: inbound 6379 only from sg-app
Important: database and Redis must never be public.

Chapter 3: EC2 and Application Setup

Web host: t3.micro

App host: minimum t3.small, preferred t3.medium

Do not use t3.micro for serious live-validation workloads.

Chapter 4: PostgreSQL and Data Layer

Recommended database: Amazon RDS PostgreSQL 16.

  • db.t3.small minimum
  • private subnets only
  • public access disabled
  • Multi-AZ for production
  • automated backups enabled

Example connection string:

postgresql+psycopg2://vinayak_app:YOUR_PASSWORD@YOUR_RDS_ENDPOINT:5432/vinayak

Example environment file:

DATABASE_URL=postgresql+psycopg2://vinayak_app:YOUR_PASSWORD@YOUR_RDS_ENDPOINT:5432/vinayak REDIS_URL=redis://YOUR_REDIS_ENDPOINT:6379/0 DHAN_CLIENT_ID=your_dhan_client_id DHAN_ACCESS_TOKEN=your_dhan_access_token TELEGRAM_TOKEN=your_telegram_token TELEGRAM_CHAT_ID=your_telegram_chat_id LIVE_TRADING_ENABLED=false TRADING_BROKER_MODE=PAPER

Chapter 5: Nginx and Systemd

Nginx example:

server { listen 80; server_name vinayak.yourdomain.com; location / { proxy_pass http://127.0.0.1:8000; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; } }

Systemd example:

[Unit] Description=Vinayak FastAPI After=network.target [Service] User=ubuntu WorkingDirectory=/opt/Trading EnvironmentFile=/etc/vinayak/vinayak.env ExecStart=/opt/Trading/venv/bin/python -m uvicorn vinayak.api.main:app --host 0.0.0.0 --port 8000 Restart=always RestartSec=5 [Install] WantedBy=multi-user.target

Chapter 6: Verification and Operations

curl http://127.0.0.1:8000/health ss -tulpn | grep 8000 sudo journalctl -u vinayak -n 100 --no-pager sudo journalctl -u nginx -n 100 --no-pager

Deployment order:

  1. Create RDS
  2. Create Redis if needed
  3. Launch app EC2
  4. Install dependencies
  5. Configure env file
  6. Create systemd service
  7. Verify 127.0.0.1:8000
  8. Configure Nginx
  9. Connect ALB
  10. Point Route 53

Chapter 7: Sizing and Cost Planning

  • Lean validation stack: about USD 45 to 90 per month
  • Stronger validation stack: about USD 80 to 160 per month
  • Production-oriented baseline: about USD 180 to 400 per month plus transfer and storage growth

Chapter 8: Rollout Phases

  1. Controlled paper validation
  2. Small live validation
  3. Hardened pre-production
  4. Stronger production posture

Final Recommendation

For Vinayak, the best near-term AWS production shape is:

  • separate web page tier
  • separate private application tier
  • separate private PostgreSQL database
  • optional Redis
  • CloudWatch and Secrets Manager
  • phased rollout from paper to tiny live validation

This is the right structure for Vinayak if you want it to behave like a serious trading platform rather than a single-host experiment.
