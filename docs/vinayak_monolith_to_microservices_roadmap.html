Vinayak Monolith To Microservices Roadmap

A practical path for evolving Vinayak from a modular monolith into selective microservices without destabilizing trading operations.

Purpose

This roadmap explains how Vinayak should evolve from its current modular monolith into a selective microservices architecture. The recommended near-term position is to keep the platform modular, deploy it cleanly in a 3-tier AWS shape, and extract only the services that produce clear operational value.

Current State

Vinayak is not a microservices platform today. It is a modular monolith because the backend still runs as one deployable application, most domains communicate in-process, and data ownership remains mostly centralized.

Why Not Split Everything Now

Premature microservices would add complexity in deployments, monitoring, retries, networking, security, and incident debugging. For Vinayak, those costs are not justified until the operational boundaries are proven to matter.

Best Near-Term Architecture

  • separate web tier
  • separate private app tier
  • separate private PostgreSQL database
  • optional Redis
  • CloudWatch and Secrets Manager
  • modular monolith application

Service Extraction Principles

A Vinayak module should become a service only if it needs independent scaling, a different reliability profile, a distinct deployment cadence, a clear business boundary, or it causes repeated operational friction inside the monolith.

Good Service Candidates

Observability Service

A strong first candidate because it has lower business risk than execution and has a clean separation from core trade execution. It should own metrics aggregation, alerts, health summaries, and dashboard payloads.

Notification Service

An excellent early extraction because it is naturally asynchronous, easy to model as events, and has limited data ownership. It should own Telegram delivery and operational notifications.

Execution Service

A later candidate, not an early one. It is highly sensitive and should only be extracted after paper-mode stability, tiny live validation, idempotency, audit logs, and kill-switch behavior are fully proven.

Strategy Runtime Service

A late-stage candidate for cases where strategy workloads need distinct scaling or independent rollout cadence. It should not be extracted before the platform proves that separation is necessary.

Areas To Keep Central Longer

  • authentication
  • database core ownership
  • configuration and secrets management

Recommended Extraction Order

Phase 0. Stabilize The Modular Monolith

Finish structure cleanup, keep runtime stable, validate paper workflows, and verify observability freshness.

Phase 1. Add Clear Internal Boundaries

Define typed interfaces and reduce cross-module leakage so the monolith behaves like future services internally.

Phase 2. Introduce Event Boundaries

Publish internal events and move non-critical side effects out of synchronous request paths.

Phase 3. Extract Notification Service

The safest first extraction. Low risk, small blast radius, and immediate value.

Phase 4. Extract Observability Service

The next best extraction. It improves operations without directly endangering broker execution.

Phase 5. Harden Execution Boundaries

Paper and tiny live validation must be stable before considering execution extraction.

Phase 6. Extract Execution Service

Only after dedicated credentials, idempotency, replay protection, and alerting are dependable.

Phase 7. Evaluate Strategy Runtime Extraction

Do this only if research and production workloads genuinely need separate scaling or deployment speed.

Practical Recommendation

  1. Keep the modular monolith now.
  2. Finish the 3-tier AWS deployment.
  3. Validate paper mode thoroughly.
  4. Validate tiny live mode cautiously.
  5. Introduce event boundaries.
  6. Extract notifications first.
  7. Extract observability second.
  8. Reassess execution extraction only after live stability.

Final Position

Vinayak should not become microservices everywhere. It should become a clean modular monolith first, then a selectively extracted platform later, with reliability-first trading behavior at every step.
