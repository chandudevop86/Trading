Architecture

TRADING APP ARCHITECTURE DOCUMENTATION

Overview
This Trading App is designed with multiple tiers so market data, strategy logic, execution flow, notifications, and reporting stay organized and easy to manage.

WEB OR UI TIER
Usually UI tier is the one which has frontend and user interaction components.
In this project, UI tier is mainly Streamlit dashboard.
This tier is used for symbol selection, strategy selection, risk setup, lot size setup, execution mode, charts, signal review, and trade analysis.
If required in production, this UI can be exposed through Nginx reverse proxy.

APP OR STRATEGY TIER
App tier is the main business logic layer.
In this project it contains market data fetch, technical strategy execution, signal generation, strike selection, futures contract mapping, execution preparation, and reporting flow.
Major modules include Trading.py, live_ohlcv.py, breakout_bot.py, supply_demand.py, indicator_bot.py, one_trade_day.py, mtf_trade_bot.py, execution_engine.py, auto_backtest.py, and auto_run.py.
Usually this tier should not be directly open to the internet.

MARKET DATA TIER
Market data tier provides the raw inputs required for strategy calculations.
This project uses Yahoo Finance OHLCV data, NSE option chain data, NSE futures derivative data, and uploaded CSV files for testing or backtesting.
This tier is used for live candles, spot price, option metrics, futures contract data, and historical analysis.

EXECUTION TIER
Execution tier handles paper execution and live-execution-ready flow.
This tier converts trade signals into executable candidate rows, normalizes quantity using lot size, and writes execution logs.
Default operation uses PAPER flow, while LIVE broker routing is optional and should be enabled only after deliberate setup.
Broker adapters can be attached here for future deployments that require real order routing.

STORAGE TIER
Storage tier keeps the application outputs and logs.
This project mainly uses CSV, YAML, HTML, and PDF files.
Examples include live_ohlcv.csv, paper_trading_logs_all.csv, live_trading_logs_all.csv, backtest_results_all.csv, nifty_options_scored.csv, and nifty_futures_snapshot.csv.

NOTIFICATION TIER
Notification tier is used for communicating signals and summaries.
In this project Telegram is used for alerts, signal summaries, and report delivery.

REPORTING TIER
Reporting tier generates user-readable outputs.
This includes CSV reports, HTML summaries, PDF files, backtest summaries, and trade result snapshots.

COMMUNICATION FLOW
1. User opens the dashboard.
2. UI sends strategy settings to app tier.
3. App tier fetches market data from Yahoo or NSE sources.
4. Strategy modules generate BUY, SELL, or HOLD signals.
5. Contract layer converts signals into options or futures contracts.
6. Execution tier prepares paper or live-ready trade rows.
7. Storage tier writes logs and result files.
8. Notification tier sends Telegram alerts if enabled.
9. Reporting tier prepares summaries and downloadable documents.

DEPENDENCY FLOW
UI Tier depends on App Tier.
App Tier depends on Market Data Tier.
Execution Tier depends on App Tier.
Storage Tier depends on App Tier and Execution Tier.
Notification Tier depends on App Tier and Execution Tier.
Reporting Tier depends on Storage Tier and App Tier.

Deployment

TRADING APP DEPLOYMENT DOCUMENTATION

Overview
This document explains how the Trading App can be deployed for local use, server use, and production-style access.

LOCAL DEPLOYMENT
The project can run locally using Python and Streamlit.
Install dependencies from requirements.txt.
Launch the dashboard using Streamlit.
This mode is suitable for development, testing, and strategy review.

APPLICATION ENTRY POINTS
Main UI entry point is Trading.py.
Other entry points include auto_backtest.py, auto_run.py, nifty50.py, nifty_options.py, and nifty_futures.py.
These scripts support automation, scoring, and reporting outside the UI.

WEB ACCESS LAYER
For production-style deployment, Streamlit can be placed behind Nginx.
Nginx can work as reverse proxy for domain-based access.
This improves routing and gives a cleaner public endpoint.

SERVER RECOMMENDATION
Recommended server setup:
1. Linux or Windows VM.
2. Python runtime.
3. Streamlit service.
4. Optional Nginx reverse proxy.
5. Secure secrets storage for tokens and credentials.

CONFIGURATION REQUIREMENTS
Required configuration may include Telegram bot token, Telegram chat ID, broker credentials for future live trading, and optional AWS S3 configuration.
These values should be stored in environment variables, secrets, or protected config files.

DATA AND FILE PATHS
Application reads and writes under the data folder.
Make sure the deployment host has read and write permission for logs, report files, and snapshots.
If using persistent server deployment, this folder should be backed up or mounted on persistent storage.

SECURITY RECOMMENDATIONS
Only UI tier should be internet-facing.
App and execution internals should remain private.
Do not expose secrets inside source files.
Use HTTPS when publishing through a domain.
Protect broker APIs and notification credentials.

AUTOMATION OPTIONS
The project already supports scheduled or repeated runs using scripts and scheduler folders.
Auto backtests and reporting tasks can be scheduled using Windows Task Scheduler, cron, or container orchestration.

DEPLOYMENT MODES
Development Mode:
Run Streamlit locally for testing.

Internal Server Mode:
Run Streamlit on internal network for team use.

Production Mode:
Run Streamlit behind Nginx with restricted access, secrets management, and scheduled automation.

FUTURE EXTENSIONS
The deployment design allows addition of:
Broker API execution.
Container deployment.
Kubernetes deployment.
Cloud object storage for reports.
Monitoring and alerting.

Communication Notes
The dashboard does not directly execute trades by itself.
It passes user-selected settings into the application logic layer.
The application logic layer fetches market data, runs strategies, maps contracts, and prepares execution rows.
The execution engine stores output into log files.
The reporting layer converts those outputs into readable summaries.
The notification layer shares important signals through Telegram.

Security Notes
UI can be exposed with controlled access.
Execution internals and secrets should remain protected.
Broker integrations should be attached only inside secure backend flow.

Runbook

TRADING APP PRODUCTION RUNBOOK

Purpose
This runbook explains how to operate, monitor, and troubleshoot the Trading App in a production or semi-production environment.

STARTUP CHECKLIST
1. Verify Python environment is available.
2. Verify required packages are installed.
3. Verify Streamlit app starts successfully.
4. Verify data folder is writable.
5. Verify Telegram credentials are configured if alerts are needed.
6. Verify broker credentials are configured if future live execution is enabled.

RUNTIME CHECKS
1. Confirm dashboard is reachable.
2. Confirm OHLCV data is loading.
3. Confirm strategy outputs are generated.
4. Confirm paper execution logs are updating.
5. Confirm report files are being created.
6. Confirm Telegram alerts are delivered when enabled.

DAILY OPERATIONS
Review live candles and trade signals.
Check paper or live trading logs.
Check backtest summary outputs.
Archive important CSV and PDF reports.
Validate that scheduled tasks completed without failure.

INCIDENT SCENARIOS
No market data:
Check external data provider access and symbol formatting.

No trade signals:
Check strategy settings, timeframe, and available candles.

Execution log not updating:
Check write permission on data directory and execution mode.

Telegram not sending:
Check token, chat ID, internet access, and notifier settings.

Wrong contract mapping:
Check whether Options or Futures mode is selected.
Check strike step, moneyness, and symbol normalization.

BACKUP RECOMMENDATION
Back up data folder regularly.
Critical files include paper trading logs, live trading logs, backtest summaries, and generated reports.

CHANGE MANAGEMENT
Apply changes first in local or testing environment.
Validate strategies and outputs.
Then promote to production server.
Avoid editing live configuration without backup.

FUTURE OPERATIONS IMPROVEMENT
Add service monitoring.
Add broker health checks.
Add log rotation.
Add alerting for failed scheduled runs.
Add persistent cloud storage for reports.

Safety Note

KRSH is positioned as a generic platform template. Analysis and paper workflows are the default path, and live execution remains an explicitly controlled extension.
