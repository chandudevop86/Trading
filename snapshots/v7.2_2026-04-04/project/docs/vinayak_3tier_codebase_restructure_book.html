Executive Summary

Vinayak should not only be deployed as a 3-tier platform. It should also be organized as one in the repository.

That means separating:

  1. Web concerns
  2. Application and domain logic
  3. Infrastructure and deployment assets

If the AWS deployment is separated but the repository is still mixed together, the system continues to behave like a monolith in practice.

Why Repo Separation Matters

  • changes become harder to reason about
  • deployment boundaries remain blurry
  • infra and application concerns remain mixed
  • testing boundaries stay unclear
  • ownership is weaker than it should be
A serious 3-tier platform should be separated both in runtime deployment and in repository structure.

Target Repository Shape

vinayak-platform/ web/ app/ infra/ docs/ tests/ data/ snapshots/

This creates clean top-level separation:

  • web/ for public-facing UI and reverse-proxy assets
  • app/ for application logic and domain modules
  • infra/ for deployment and environment assets
  • docs/ for architecture and operations guidance
  • tests/ for organized testing layers

Tier Responsibilities

web/

  • Nginx config
  • static assets
  • templates
  • simple frontend helpers

Must not contain trading logic, DB models, or broker code.

app/

  • FastAPI app
  • execution logic
  • risk logic
  • observability
  • messaging
  • workers
  • DB access layer
  • broker adapters

infra/

  • AWS infrastructure code
  • Terraform or CloudFormation
  • systemd units
  • nginx deployment config
  • Docker and Compose assets
  • deployment scripts

docs/ and tests/

docs/ owns architecture and runbooks. tests/ owns unit, integration, and end-to-end verification.

Current-To-Target Migration Map

  • vinayak/api → app/vinayak/api
  • vinayak/execution → app/vinayak/execution
  • vinayak/observability → app/vinayak/observability
  • vinayak/messaging → app/vinayak/messaging
  • vinayak/analytics → app/vinayak/analytics
  • vinayak/metrics → app/vinayak/metrics
  • vinayak/notifications → app/vinayak/notifications
  • deploy/ → infra/
  • vinayak/tests → tests/unit, tests/integration, tests/e2e

Proposed Directory Tree

F:/Trading web/ nginx/ static/ templates/ README.md app/ vinayak/ api/ analytics/ core/ db/ execution/ messaging/ metrics/ notifications/ observability/ validation/ workers/ __init__.py requirements.txt alembic.ini README.md infra/ aws/ docker/ nginx/ scripts/ systemd/ README.md docs/ architecture/ aws/ runbooks/ validation/ tests/ unit/ integration/ e2e/ data/ snapshots/

Migration Phases

  1. Documentation-first separation
  2. Infra extraction
  3. App namespace consolidation
  4. Test reorganization
  5. Web extraction

This keeps the restructuring controlled instead of turning it into a risky big-bang rewrite.

Naming And Ownership Rules

  1. web/ owns presentation and proxy assets only
  2. app/ owns business logic and runtime logic only
  3. infra/ owns environment and deployment assets only
  4. docs/ owns architecture and runbook material only
  5. tests/ mirrors behavior verification, not deployment concerns
If a file mixes concerns, split it before moving it.

Final Recommendation

If Vinayak should look and behave like a serious 3-tier platform, the repository must reflect that separation clearly.

The correct target is:

  • separate web/
  • separate app/
  • separate infra/
  • structured docs/
  • structured tests/

This improves maintainability, ownership, deployment discipline, and long-term production safety.
