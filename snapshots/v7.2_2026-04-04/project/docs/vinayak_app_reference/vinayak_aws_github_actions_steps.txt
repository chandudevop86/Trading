Vinayak AWS Implementation Guide 3
GitHub Actions Pipeline For ECR And ECS

Goal
Create a CI CD pipeline that builds the Vinayak image, pushes it to ECR, and deploys it to ECS.

Scope
1. GitHub Actions workflow
2. AWS authentication
3. Docker build and tag
4. ECR push
5. ECS task definition render
6. ECS service deploy
7. Optional migration step

Recommended repository files
.github/workflows/deploy-vinayak.yml
infra/app/aws/ecs-task-definition.json

Step 1
Prepare AWS side first.
Create ECR repository vinayak-api.
Create ECS cluster and service.
Create task definition family vinayak-api.
Create IAM permissions for GitHub deployment.

Step 2
Choose authentication method.
Preferred:
GitHub OIDC to AWS IAM role.
Avoid long-lived AWS keys if possible.
Create an IAM role trusted by GitHub for your repo and branch.
Grant it ECR push and ECS deploy permissions.

Step 3
Store GitHub secrets or variables.
If using OIDC, keep secrets minimal.
Use variables for AWS region, ECR repo name, ECS cluster, ECS service, and task definition path.
If not using OIDC, store AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY as encrypted secrets.

Step 4
Create workflow triggers.
Typical trigger:
on push to main.
Optional manual trigger with workflow_dispatch.
Optional separate deploy-dev and deploy-prod workflows.

Step 5
Build stage.
Checkout repo.
Set up Docker buildx.
Configure AWS credentials.
Login to ECR.
Build image from vinayak Dockerfile.
Tag with commit SHA and latest.
Push both tags to ECR.

Step 6
Render ECS task definition.
Use the latest image URI.
Either:
1. template the task definition JSON inside the workflow, or
2. keep a base JSON in the repo and replace only the image field.
Update environment or secret refs only if needed.

Step 7
Deploy to ECS.
Register the new task definition revision.
Update the ECS service to use that revision.
Wait for the service to stabilize.
Fail the workflow if deployment stabilization fails.

Step 8
Add database migration step.
Preferred approach:
run alembic migration as a one-off ECS task before switching traffic.
Alternative:
run migration from a GitHub runner only if RDS network access allows it.
Safer production pattern is one-off task inside the VPC.

Step 9
Add health verification.
After deploy, hit /health/live.
Then hit /health/ready.
Optionally check /workspace with a smoke test if auth/session handling is available.

Step 10
Suggested workflow stages.
1. lint or syntax checks.
2. build image.
3. push image to ECR.
4. render task definition.
5. deploy to ECS.
6. wait for service stability.
7. smoke test.

Step 11
Suggested GitHub Actions actions.
aws-actions/configure-aws-credentials
aws-actions/amazon-ecr-login
aws-actions/amazon-ecs-render-task-definition
aws-actions/amazon-ecs-deploy-task-definition
actions/checkout

actions/setup-python if you want Python checks first.

Step 12
Permissions the deployment role needs.
ECR:
GetAuthorizationToken.
BatchCheckLayerAvailability.
PutImage.
InitiateLayerUpload.
UploadLayerPart.
CompleteLayerUpload.

ECS:
DescribeServices.
DescribeTaskDefinition.
RegisterTaskDefinition.
UpdateService.
ListTasks.
DescribeTasks.

IAM pass role permission for the ECS execution and task roles used by the task definition.
CloudWatch read permissions are useful for debugging.

Step 13
Branch strategy.
Use main for production deploys.
Use develop for a staging ECS service if needed.
Keep environment specific variables separate.
Protect the production environment with manual approval in GitHub environments.

Step 14
Failure handling.
If deployment fails, do not overwrite the old stable image tag manually.
Let ECS keep the previous healthy task definition revision available.
Record the failing revision and inspect CloudWatch logs.

Step 15
Later upgrades.
Add Terraform plan checks before deploy if you also use Terraform.
Add image vulnerability scanning gates.
Add migration approval step for production.
Add Slack or Telegram deploy notifications.

What you should implement later in code and ops
1. Add a proper Docker image tag strategy.
2. Run tests before build.
3. Add one-off migration tasks.
4. Add separate staging and production ECS services.
5. Add manual approval for production GitHub environment.
