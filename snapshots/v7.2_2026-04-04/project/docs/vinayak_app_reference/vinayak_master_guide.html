Part 1: AWS Architecture

This section explains the target AWS design for Vinayak and how your current trading project evolves into a cloud-ready multi-tier platform.

Current Starting Point

  • Python trading application with Streamlit UI
  • Strategy logic and execution flow in the same application layer
  • Dhan broker integration and Telegram notifications
  • Existing deployment guidance for Nginx, Docker, and Kubernetes

Target AWS Architecture

Only the web tier should be internet-facing. The app tier, cache, and database tiers should remain private inside the VPC.

Part 2: AWS Interview Questions and Answers

1. What is the current architecture of your project?

The current project is mainly a Python trading application with a Streamlit-based UI and business logic in the same layer. It already includes broker integration, notifications, and deployment guidance, making it a good candidate for AWS modernization.

2. How would you move this project to AWS?

I would start with EC2 and Nginx for a simple first deployment, then add RDS, Redis, S3, and queue-based processing. After that I would move toward a cleaner service separation using the Vinayak architecture.

3. Why should the app tier not be public?

Because it contains business logic, broker integration, and sensitive operations. Best practice is to expose only the web tier publicly and keep internal services private behind security groups and subnet controls.

4. Why use RDS PostgreSQL here?

The platform has relational entities like signals, trades, executions, settings, and logs. PostgreSQL fits that model well and RDS reduces database operational work.

5. Where does Redis help?

Redis helps cache option-chain data, hot market snapshots, and short-lived live state to reduce repeated external fetches and improve response speed.

6. EC2 or ECS first?

I would use EC2 first because it is easier to deploy and explain. Once the service boundaries are cleaner, I would move to ECS for easier scaling and operations.

7. How would you secure secrets?

I would use Secrets Manager or SSM Parameter Store and attach IAM roles to the compute tier instead of hardcoding secrets in code or permanent env files.

8. Why add SQS or MQ?

To move notifications and slow trade-processing workflows off the request path, which improves responsiveness and reliability.

9. How would you monitor the platform?

I would use CloudWatch for logs, metrics, and alarms, and keep app-level audit logging for reviewed trades, executions, and broker payloads.

10. How would you summarize this project in an interview?

I would describe it as a Python trading platform evolving from a monolith into a cloud-ready AWS multi-tier architecture with clear web, app, database, cache, and messaging responsibilities.

Part 3: AWS Deployment Roadmap

Stage 1: Single-Server Deployment

Deploy the stable app on EC2, put Nginx in front, attach Route 53, and enable HTTPS.

Stage 2: Persistence

Introduce RDS PostgreSQL and move important state from local files and session state into the database. Use S3 for exports and reports.

Stage 3: Cache

Add ElastiCache Redis for option-chain cache, hot OHLCV data, and live reviewed state.

Stage 4: Async Workflows

Use SQS or Amazon MQ for alerts, background processing, and execution requests.

Stage 5: Containerization

Package Vinayak into Docker images and move to ECS when the modules are stable and clearly separated.

Recommended order: protect v3, deploy on EC2, add Route 53 and HTTPS, move data to RDS, add S3, then Redis, then queues, then ECS.
